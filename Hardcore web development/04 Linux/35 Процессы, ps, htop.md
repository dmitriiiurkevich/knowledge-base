
*Процесс* — это абстракция операционной системы, которая позволяет операционной системе выделять процессорное время и память компьютера для выполнения нескольких программ.

С точки зрения операционной системы каждый процесс состоит из трёх частей:
1. исполняемой программы,
2. данных исполняемой программы,
3. контекста процесса

*Программа* — это последовательность операций над данными, загруженная в оперативную память. 

*Данные программы* располагаются в оперативной памяти и на внешних устройствах. Перед выполнением очередной операции программы все необходимые для этого данные должны быть загружены в оперативную память компьютера.

Каждая программа выполняется в отдельном виртуальном адресном пространстве, поэтому процессы защищены друг от друга, и аварийное завершение одной программы никак существенно не повлияет как на остальные выполняющиеся программы, так и на всю систему в целом.

*Контекст процесса* позволяет операционной системе отложить процесс на некоторое время, а затем вернуться к нему, когда это будет возможно. Контекст процесса, например, может содержать следующие данные:
- состояние регистров процессора;
- состояние стека программы;
- состояние стека операционной системы;
- состояния зарезервированной памяти.

Контекст необходим для того, чтобы можно было реализовать 
- работу *диспетчера операционной системы*, 
- обеспечить межпроцессное взаимодействие (*IPC* — *Inter Process Communication*) 
- обмен информацией между процессами и операционной системой.

У каждого процесса есть свой идентификатор — *process id*, *pid*.

Все процессы объединены в
- *process groups* — группы процессов
- *process sessions* — сессии процессов (их ещё иногда называют сеансами).

Процессы, принадлежащие к одной группе, определяются общим идентификатором группы — *group id*, *gid* — идентификатором процесса, который создал эту группу — т. н. процесса *лидера группы*.

*Group leader process* — это процесс, у которого *pid = gid*.

Процессы, принадлежащие к одной сессии, определяются общим идентификатором сессии — *session id*, *sid* — идентификатором процесса, который создал эту сессию — т. н. процесса  *лидера сессии*.

*Session leader process* — это процесс, у которого *pid = gid = sid*.

В каждой сессии есть особая группа процессов, называемая *терминальной группой* — *foreground group*. Все остальные группы процессов сессии называются *background groups*. 

В каждой сессии доступ к потоку ввода предоставляется только процессам из *foreground group*, и обычно только одному из них — *session leader* процессу — запуск которого блокирует терминал.

Ядро предоставляет системные вызовы как для управления процессами, так и для создания новых процессов:
- *fork* — создаёт новое виртуальное адресное пространство, полностью идентичное виртуальному адресному пространству родительского процесса. После выполнения этого системного вызова мы получаем два абсолютно одинаковых процесса: родительский и дочерний. Функция *fork* возвращает 
	- 0 в дочернем процессе
	- *pid* дочернего процесса в родительском процессе.
- *exec* — запускает в виртуальном адресном пространстве программу и передаёт запущенной программе параметры её вызова.
- *wait* — заставляет родительский процесс ожидать завершения всех его дочерних процессов.
- *kill* — отправляет сигнал выбранному процессу.
- nice — изменяет приоритет выполняемого процесса. 


Любая программа может начать выполняться, только в одном из дыух случаев:
1. ее выполнение запустил какой-то другой процесс, 
2. произошло какое-то прерывание (например, прерывание внешнего устройства).

В связи с развитием *SMP* *(Symmetric Multiprocessor Architectures)* в ядро *Linux* был внедрен механизм тредов — *threads* — или потоков управления. Все треды одного процесса выполняются в виртуальной памяти этого процесса, и совместно используют эту виртуальную память.

Если интерпретатору *shell* встречается команда, соответствующая некоторому выполняемому файлу, интерпретатор выполняет эту коанду, начиная с так называемой точки входа — *entry point*. Например, точкой входа для программы, написанных на языке  *С*, будет функция *main*. Запущенная программа может создать дочерний процесс, т.е. запустить какую-то другую программу, выполнение которой тоже начнется с запуска функции *main*.

`$ pstree` — выведет дерево процессов.

`$ kill my_process_pid` — отправляет сигнал выбранному процессу.

`$ killall my_processname` — отправляет *SIGTERM* всем процессам с именем *my_processname*.

`$ sudo killall my_processname` — отправляет *SIGTERM* всем процессам с именем *my_processname*, запущенным от имени любого пользователя.


`$ sudo pkill ...` — отправляет сигнал всем процессам, удовлетворяющим критериям поиска.

`$ sudo pkill -9 -u my_username`  — отправляет *SIGKILL* всем процессам, запущенным от имени пользователя *my_username*.
## ps

По умолчанию *ps* возвращает список процессов 
- *euid* которых = *uid* текущего пользователя,
- ассоциированных с (*associated with*) тем же терминалом, в котором была запущена команда. 

Список процессов отображается в виде таблицы со следующими колонками:
- *PID* — *pid* — идентификатор процесса
- *TTY* — *tname* — терминал, ассоциированный с процессом
- *TIME* — *time* — cumulated *CPU* time
- *CMD* — *ucmd* — команда или имя исполняемого файла


`ps -A`, `ps -e` — вернёт список вообще всех процессов.

`ps -a` — вернёт список всех процессов, ассоциированных с текущим терминалом, кроме лидеров сессий.

`ps -d` — вернёт список всех процессов кроме лидеров сессий.
`ps -d -N`, `ps -d --deselect` — инвертирует предыдущий список — вернёт список процессов лидеров сессий.

`ps -G my_ps_group_id`
`ps -S my_ps_session_id`

`ps -C my_command` — вернёт список всех процессов *my_command*.
`ps -U my_user_name` — вернёт список всех процессов, запущенных от имени *my_user_name*.
