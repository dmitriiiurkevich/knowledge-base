
## 1. User Space (виртуальная память процесса)

Нижняя часть адресного пространства. 
Процесс имеет полный доступ ко всем разделам User Space.

### 1.1. text section
- содержит исполняемый код программы
- в течение всего времени жизни процесса
    - размер и содержимое не изменяются
    - доступен только для чтения
- шарится между всеми экземплярами исполняемой программы

### 1.2. data section
- содержит глобальные и статические переменные исполняемой программы
- в течение всего времени жизни процесса
    - имеет фиксированный размер, но не содержимое
    - доступен для чтения и записи
- переменные, проинициализированные и неинициализированные при создании, хранятся в двух разных подразделах:
    - **initialized data segment (.data)**
        - `int count = 42;`
        - `char *message = "Hello World";` — указатель здесь, сама строка в .rodata
    - **uninitialized data segment (.bss)**
        - `int buffer[1000];`
        - `static char name[50]`
        - операционная система выделяет необходимую память и заполняет её нулями
        - _BSS_ — _Block Started by Symbol_ — историческое название
- уникален для каждого процесса
- шарится между всеми потоками в процессе

### 1.3. rodata section (read-only data)
- содержит read-only данные, которые не должны изменяться во время выполнения: строковые литералы, массивы констант
- доступен только для чтения
- может шариться между процессами
- отделён от .data для защиты от случайной записи

### 1.4. stack
- содержит
    - локальные переменные функций
    - **return addresses** — адреса памяти, которые указывают, где должно продолжаться выполнение программы после завершения функции — по одному адресу на каждый уровень стека вызовов
    - **saved registers** — копии значений регистров CPU, которые необходимо сохранить перед вызовом функции и восстановить после её завершения
- дополнительная память автоматически аллоцируется/деаллоцируется при вызове/завершении работы функции
- LIFO
- имеет ограничения, конфигурируемые на уровне ОС
    - maximum size limit — значение по умолчанию:
    - 8 MB в Linux 64-bit, 
    - 4 MB в Windows 64-bit, 
    - 1 MB в Windows 32-bit
- динамически изменяет свой размер в пределах ограничений (и может переполняться — stack overflow)
- уникален для каждого процесса
- уникален для каждого потока в процессе

### 1.5. heap
- содержит
    - объекты/данные, размер которых не известен на момент компиляции
    - данные, которые необходимо сохранить за пределами области видимости создавшей их функции
- ограничен объёмом виртуальной памяти, максимальный объём виртуальной памяти, доступный одному процессу, задаётся праметром операционной системы ulimit, 
- динамическая аллокация памяти:
    - программа вызывает malloc()/free() — библиотечные функции
    - malloc() внутри использует системные вызовы brk/sbrk или mmap для получения памяти от ядра
    - malloc держит свой пул — нарезает большие куски от ядра на мелкие блоки для программы
    - free() возвращает блок в пул malloc'а, но не обязательно отдаёт память обратно ядру
- возможны проблемы с фрагментацией памяти
- уникален для каждого процесса
- шарится между всеми потоками в процессе

### 1.6. другие разделы
- **memory-mapped files (mmap)** — файлы, отображённые в память
- **shared libraries** — код разделяемых библиотек (.so / .dll)
- **shared memory** — разделы для межпроцессного взаимодействия (IPC)
- **vDSO** (virtual dynamic shared object) — код ядра, доступный из user space для ускорения системных вызовов

## 2. Kernel Space

Верхняя часть адресного пространства процесса, 
- не достуна процессу напрямую
- доступна только через системные вызовы с переключением в kernel mode.
- содержит данные, которые ядро хранит про каждый процесс. В Linux — для этого используется специальная структура `task_struct`.

### 2.1. Идентификация процесса
- **PID** — process id
- **PPID** — parent process id
- **TGID** — thread group id (для потоков)
- **Session ID**, **Process Group ID** — для управления job control

### 2.2. Состояние процесса (process execution state)
- **Running (R)** — выполняется или готов к выполнению
- **Interruptible sleep (S)** — ждёт события, можно разбудить сигналом
- **Uninterruptible sleep (D)** — ждёт I/O, нельзя прервать (например, диск)
- **Stopped (T)** — остановлен сигналом (SIGSTOP, SIGTSTP)
- **Zombie (Z)** — завершился, но родитель не забрал exit code через wait()

### 2.3. Memory management structures
- **page tables** — таблицы трансляции виртуальных адресов в физические
- **memory maps (vm_area_struct)** — какие разделы памяти выделены, с какими правами
- **memory limits** — ограничения на использование памяти (RSS, virtual)
- **mm_struct** — главная структура, описывающая адресное пространство

### 2.4. File descriptor table
- массив открытых файлов/сокетов/пайпов
- stdin (0), stdout (1), stderr (2) — первые три по умолчанию
- каждый fd указывает на file structure в ядре
- лимиты: ulimit -n (обычно 1024 по умолчанию)

### 2.5. Scheduling information
- **priority** — приоритет (nice value: -20..19, real time priority)
- **scheduling policy** — SCHED_NORMAL, SCHED_FIFO, SCHED_RR и др.
- **timeslices** — сколько времени CPU осталось в текущем кванте
- **CPU affinity** — на каких ядрах можно выполняться
- **cgroups** — ограничения ресурсов для группы процессов

### 2.6. Security context
- **UID / EUID** — реальный и эффективный user id
- **GID / EGID** — реальный и эффективный group id
- **capabilities** — гранулярные права (CAP_NET_BIND_SERVICE, CAP_SYS_ADMIN и др.)
- **SELinux/AppArmor context** — дополнительные ограничения безопасности
- **seccomp** — фильтрация системных вызовов

### 2.7. Signal handling
- **signal handlers** — таблица обработчиков для каждого сигнала
- **pending signals** — сигналы в очереди ожидания
- **blocked signals (signal mask)** — какие сигналы заблокированы

### 2.8. Resource usage & accounting
- **CPU time consumed** — user time, system time
- **memory usage** — RSS, virtual memory
- **I/O statistics** — bytes read/written
- **context switches** — voluntary, involuntary

### 2.9. Namespaces (nsproxy)
- **PID namespace** — изоляция PID'ов, процесс видит только процессы в своём namespace
- **Network namespace** — свой сетевой стек (интерфейсы, routing tables, iptables)
- **Mount namespace** — своя файловая система (что куда примонтировано)
- **UTS namespace** — свой hostname и domain name
- **IPC namespace** — изоляция System V IPC и POSIX message queues
- **User namespace** — маппинг UID/GID (можно быть root внутри, но не снаружи)
- **Cgroup namespace** — изоляция видимости cgroups
- **Time namespace** (Linux 5.6+) — своё системное время

Каждый процесс имеет указатель на nsproxy — структуру со ссылками на все его namespaces. При создании нового процесса с помощью `clone()` можно унаследовать namespaces от родителя или создать новые через
- `clone()` с флагами CLONE_NEW* или 
- `unshare()`.

### 2.10. Связи процесса с другими процессами
- **parent pointer** — указатель на родительский процесс
- **children list** — список дочерних процессов
- **thread list** — список потоков в процессе








*N.B.*
С точки зрения операционной системы каждый поток является процессом.

Однопоточный процесс = главный поток процесса.

Многопоточный процесс = группа процессов с общим TGID и разными PID. 

PPID у них у всех тоже общий.

Главный поток в многопоточном процессе  = процесс, у которго TGID и PID совпадают.


```
Процесс X (PID=50) вызывает fork() → создаётся процесс Y

Процесс Y создаёт потоки:

task_struct 1: PID=100, TGID=100, PPID=50  (главный поток)
task_struct 2: PID=101, TGID=100, PPID=50  (поток)
task_struct 3: PID=102, TGID=100, PPID=50  (поток)
```



*N.B.*
Чем fork отличается от clone?

`fork()` — создаёт копию процесса с изолированной памятью (copy-on-write).

`clone()` — создаёт копию процесса и принимает флаги, регулирующие, что дочерний процесс будет разделять с родительским —  память, файлы, namespace и т.д. — а что у дочернего процесса будет своим, изолированным от родительского процесса.

По сути,
- `fork()` — это `clone()` с набором флагов "скопировать всё" — создаёт новый полностью изолированный процесс.
- `pthread_create()` — это `clone()` с флагами "разделять всё" — создаёт новый поток в текущем процессе.


*N.B.*
Если клонировать процесс с тремя потоками, у процеса-клона тоже будет три потока?

Нет, системный вызов `fork()` клонирует только тот поток, из которого был произведён вызов. 

У нового процесса будет один поток, остальные два потока не скопируются — и это источник потенциальных проблем.

```
Родительский процесс:
  Мьютекс для синхронизации потоков
  Поток 1: держит мьютекс залоченным
  Поток 2: работает
  Поток 3: вызывает fork()

Дочернем процессе:
  Копия залоченного мьютекса
  Копия потока 3: не имеет возможности разблокировать копию мбютекса — deadlock
```


*N.B.*
У init-процесса 
- PID = 1
- PPID = 0

Процесс с PID = 0 — это фиктивный idle process, не являющийся процессом в обычном смысле этого слова.
