
Реляционная модель данных — Edgar F. Codd, 1969

## Основные понятия

1.
*Тип данных* — понятие полностью аналогичное понятию типа данных во всех прочих языках программирования, может быть встроенным или пользовательским (абстрактным).

2.
*Домен* — домены не поддерживаются в Postgres, как большинстве реляционных СУБД, но всё ещё поддерживаются в Oracle.

Интуитивно *домен* представляет собой потенциально допустимое множества значений данного типа. 

Данные считаются сравнимыми только в том случае, когда они относятся к одному *домену*. Например, значения обоих доменов "Номера пропусков" и "Номера документов" относятся к типу целых чисел, но при этом не являются сравнимыми. 

В самом общем виде *домен* определяется заданием 
- некоторого базового типа данных
- произвольного логического выражения, применяемого к элементу типа данных. 

Если вычисление этого логического выражения дает результат *True*, то элемент данных является элементом домена.


3.
*Схема отношения* — именованное множество пар {имя атрибута: тип атрибута} — или, если домены поддерживаются используемой СУБД, то {имя атрибута: домен атрибута}.

Мощность этого множества называется степенью (или *арностью*) схемы отношения.

Пример отношения четвёртой степени (кватернарного или полиадического отношения): 

``` sql
Employee (
	employee_id: INTEGER, 
	first_name: VARCHAR(50), 
	last_name: VARCHAR(50), 
	department_id: INTEGER
)
```

Схема отношения также может содержать дополнительные ограничения на хранимые данные — *констрейнты*.

4.
*Схема БД* (в структурном смысле) — это набор схем отношений.

5.
*Кортеж*, соответствующий некоторой схеме отношения, — набор именованных значений заданного типа — множество пар {имя атрибута: значение}, которое содержит одно вхождение каждого имени атрибута, принадлежащего данной схеме отношения. 

Каждое значение является допустимым значением типа данных или домена данного атрибута. 

6.
*Отношение* — это
- схема отношения
- множество *экземпляров отношения* — множество кортежей, соответствующих этой схеме отношения.

7.
*Реляционная база данных* — это
- схеме БД
- множество отношений, имена которых совпадают с именами схем отношений в схеме БД.

## Ключи

Ключ — это минимальный по включению поднабор атрибутов схемы отношения, уникально идентифицирующий каждый экземпляр отношения.

Неоднозначность 1
Вообще в некоторых случаях термин ключ используется для обозначения произвольного поднабора атрибутов, не обязательно гарантирующего уникальную идентификацию набора данных.  Например, при общей постановке задачи сортировки по ключу — *key sorting*.

В теории реляционных баз данных требование возможности уникальной идентификации по ключу присутствует.

Неоднозначность 2
Есть альтернативная система определений, и она содержит в себе одну важную уязвимость.

Согласно этой системе определений, 
- *ключ* — это любой поднабор атрибутов, значения которых однозначно определяют кортеж отношения
- *первичный ключ* — это минимальный по включению поднабор атрибутов, значения которых однозначно определяют кортеж отношения.

Для каждого отношения по крайней мере полный набор его атрибутов является ключом. Далее, аксиома выбора гарантирует нам существования первичного ключа, однако не гарантирует его единственность. При наличии у некоторого отношения нескольких первичных ключей, может возникнуть соблазн начать использовать разные первичные ключи для внешних ссылок, а это является очень плохой практикой.

Поэтому было бы более надёжным считать, что любой ключ по определению обладает свойством минимальности по включению атрибутов (аксиома выбора в этом случае по-прежнему гарантирует нам существование по крайней мере одного ключа). 

Далее из всех возможных ключей мы произвольно выбираем какой-то один, этот произвольно выбранный ключ мы называем первичным, все остальные ключи при этом становятся вторичными. И отныне только этот произвольно выбранный первичный ключ будет использоваться для внешних ссылок на отношение.

Итого, выделяют следующие виды ключей:
- По структуре ключа:
    - *simple key* — ключ, состоящий из одного поля,
    - *composite key* — ключ, состоящие из нескольких полей.
- По природе ключа:
    - *natural key* — ключ, который извлекается из реальных данных,
    - *surrogate key* — ключ, который автоматически генерируются самой СУБД специально для идентификации записи. Использование суррогатного ключа может быть обусловлено тем, что имеющийся естественный ключ 
	    - отсутствует,
	    - является длинным или составным,
	    - обладает значением, которое потенциально может измениться в течение времени жизни разрабатываемой базы данных.
- По приоритету использования ключа:
    - *primary key* — единственный ключ, используемый для ссылок на таблицу. Использование различных ключей для ссылок считается плохой практикой.
    - *secondary key* — ключи, позволяющие поддерживать уникальность поля или набора полей. Их не следует использовать для ссылок на таблицу.

Атрибуты, входящие в состав первичного ключа, называются *ключевыми атрибутами*, все прочие атрибуты называются *неключевыми атрибутами*.

*N.B.*: Следует проектировать и развивать базу данных таким образом, чтобы значение  первичных ключей не изменилось бы со временем.

## Фундаментальные свойства отношений

### Отсутствие кортежей-дубликатов
В классической теории реляционных баз данных отношение — это множество кортежей, а множество не может включать в себя несколько экземпляров одного и того же элемента.

Замечание 1
Использование суррогатных ключей в некотором смысле позволяет нарушать это свойство. Однако, *СУБД* имеют в своём распоряжении специальные инструменты, позволяющие добиваться уникальности неключевых элементов.

Замечание 2
Во многих распределённых *СУБД* допускается нарушение свойства уникальности кортежей для промежуточных отношений, порождаемых неявно при выполнении запросов. Такие отношения являются не множествами, а мультимножествами, и иногда это может привести к серьезным проблемам.

### Отсутствие упорядоченности кортежей
Отсутствие требования к поддержанию порядка на множестве кортежей отношения дает дополнительную гибкость *СУБД* при хранении баз данных во внешней памяти и при выполнении запросов к базе данных.

С помощью *SQL* инструкции *ORDER BY* можно потребовать сортировки результирующей таблицы в соответствии со значениями некоторых столбцов. Такой результат, вообще говоря, будет являться не отношением, а некоторым упорядоченным списком кортежей.

### Атомарность значений атрибутов
В классической теории реляционных баз данных значения всех атрибутов являются атомарными, поскольку среди значений *домена* не могут содержаться множества значений (отношения). Поэтому принято говорить, что в реляционных базах данных допускаются только нормализованные отношения или отношения, представленные в первой нормальной форме.

В действительности же такие типы данных, как массивы, *JSON* и *JSONB*, нарушают это свойство.

## Ссылки или внешние ключи

*Внешним ключом* или *ссылкой* — *foreign key* — называют набор атрибутов, соответствующий ключу (как правило, первичному ключу) другой или той же самой таблицы.

## Плюсы и минусы реляционной модели данных

Основной плюс: 
- строгая структура данных, единая для всех экземпляров каждого отношения.

Основные минусы: 
- даже небольшая модификация схемы отношений может привести к длительной блокировке работы *БД*. Если же каждая запись *БД* потенциально использует свою собственную схему данных, то гораздо реже возникает необходимость блокировки всех записей при изменении схемы одной из них  
- реляционные *СУБД* тяжело поддаются горизонтальному масштабированию.
