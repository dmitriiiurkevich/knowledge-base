
*SELF JOIN*  — это *INNER JOIN* таблицы с самой собой. Он же любой другой *JOIN*, поскольку таблица одна и та же.

*NATURAL JOIN* — это декартово произведение двух таблиц, отфильтрованное по совпадению значений всех одноимённых атрибутов. Бывает левым правым и пр. Не следует использовать никогда.

*JOIN LATERAL* — ключевое слово *LATERAL* может применяться вместе с любой операцией *JOIN*, например, с *INNER JOIN*, *LEFT JOIN*, *RIGHT JOIN* и т.д., и позволяет правому операнду получить доступ к столбцам левого операнда.

Пример 1

Пусть у нас есть два временных ряда: 
- температура
- влажность

``` sql
CREATE TABLE temperature(  
  ts TIMESTAMP NOT NULL,  
  city TEXT NOT NULL,  
  temperature INT NOT NULL);  
  
CREATE TABLE humidity(  
  ts TIMESTAMP NOT NULL,  
  city TEXT NOT NULL,  
  humidity INT NOT NULL);
  ```  

Измерения производятся в двух городах, результаты измерений приходят из разных источников и не обязательно синхронно.

``` sql
INSERT INTO temperature (ts, city, temperature)  
SELECT ts + (INTERVAL '60 minutes' * random()), city, 30*random()  
FROM generate_series('2022-01-01' :: TIMESTAMP, '2022-01-31', '1 day') AS ts,  
     unnest(array['Moscow', 'Berlin']) AS city;  
  
INSERT INTO humidity (ts, city, humidity)  
SELECT ts + (INTERVAL '60 minutes' * random()), city, 100*random()  
FROM generate_series('2022-01-01' :: TIMESTAMP, '2022-01-31', '1 day') AS ts,  
     unnest(array['Moscow', 'Berlin']) AS city;
```

Поэтому сджойнить две таблицы по значениям таймстампов не получится.

``` sql
SELECT t.ts, t.city, t.temperature, h.humidity  
FROM temperature AS t  
LEFT JOIN humidity AS h ON t.ts = h.ts;

-- запрос ничего не сджойнит и ничего не вернёт
```


Специально для таких случаев и придумали [*ASOF JOINS*](https://duckdb.org/docs/sql/query_syntax/from.html#as-of-joins). Их семантика следующая: 
берем строку с данными из левой таблицы, 
джойним её со строкой из правой таблицы, содержащей такую же временную метку или меньше.

``` sql
SELECT t.ts, t.city, t.temperature, h.humidity  
FROM temperature AS t  
JOIN LATERAL  
  ( SELECT * FROM humidity  
    WHERE city = t.city -- данные из внешнего запроса
    AND ts <= t.ts -- данные из внешнего запроса 
    ORDER BY ts DESC 
    LIMIT 1  
  ) AS h ON TRUE  
WHERE t.ts < '2022-01-05'

-- благодавря использованию ключевого слова LATERAL
-- позапрос может использовать данные из внешнего запроса

-- без ключевого слова LATERAL ничего не получится
-- подзапрос вернёт ошибку
```

И, забегая вперёд, проверим, что такие запросы дружат с индексами.

``` sql
-- создадим covering индекс не только 
-- который ускоряет запрос на ~13% по сравнению  
-- с простым индексом.  
CREATE INDEX humidity_idx ON humidity (ts, city) INCLUDE (humidity);  
  
-- Нужно добавить побольше данных, чтобы СУБД предпочла использовать  
-- индекс, а не сканировать закэшированные строки в памяти.  
INSERT INTO temperature (ts, city, temperature)  
SELECT ts + (INTERVAL '60 minutes' * random()), city, 30*random()  
FROM generate_series('2010-01-01' :: TIMESTAMP,  
                     '2020-12-31', '1 hour') AS ts,  
     unnest(array['Moscow', 'Berlin']) AS city;  
  
INSERT INTO humidity (ts, city, humidity)  
SELECT ts + (INTERVAL '60 minutes' * random()), city, 100*random()  
FROM generate_series('2010-01-01' :: TIMESTAMP,  
                     '2020-12-31', '1 hour') AS ts,  
     unnest(array['Moscow', 'Berlin']) AS city;  
  
-- Смотрим EXPLAIN (или EXPLAIN ANALYZE).  
EXPLAIN SELECT t.ts, t.city, t.temperature, h.humidity  
FROM temperature AS t  
LEFT JOIN LATERAL  
  ( SELECT * FROM humidity  
    WHERE city = t.city AND ts <= t.ts  
    ORDER BY ts DESC LIMIT 1  
  ) AS h ON TRUE;
```


Пример 2

Пусть у нас есть две таблицы: 
- книги 
- авторы


Следующий запрос вернёт две колонки:
- автор (при условии, что в базе есть хотя бы одна его книга),
- последняя книга автора

``` sql
select
    a.name as author,
    b.name as last_author_book
from author a
join lateral (
    select name from book
    where book.author_id = a.author_id
    order by created_at desc nulls last
    limit 1
) b on true 
order by author;
```


Следующий запрос вернёт две колонки:
- автор (без всяких условий),
- последняя книга автора

``` sql
select
    a.name as author,
    b.name as last_author_book
from author a
left join lateral ( -- left join вместо inner join в предыдущем запросе
    select name from book
    where book.author_id = a.author_id
    order by created_at desc nulls last
    limit 1
) b on true 
order by author;

-- для авторов, чьи книги отсутствуют в базе, вместо последней книги
-- вернётся null
```


Эту конкретную задачу — по каждому автору достать его последнюю книгу — можно решить и обычным подзапросом:

``` sql
select
    a.name as author,
    (select name from book
     where book.author_id = a.author_id
     order by created_at desc nulls last
     limit 1) last_author_book
from author a
order by author, last_author_book;
```


Однако, в таком такой подзапрос может вернуть только 1 строку и 1 колонку. Если использовать значение *limit* = 2 — чтобы получить для каждого автора, не одну, а две его последние книги — запрос перестанет работать.

``` sql
select
    a.name as author,
    (select name from book
     where book.author_id = a.author_id
     order by created_at desc nulls last
     limit 2) last_author_book -- нельзя поместить два значения в одну колонку
from author a
order by author, last_author_book;
```

 Чтобы вернуть несколько строк из подзапроса, мы должны 
 - переместить подзапрос в блок *FROM*
 - использовать ключевое слово *LATERAL*

``` sql
select
    a.name as author,
    b.name as last_added_book
from author a
left join lateral (
    select name from book
    where book.author_id = a.author_id
    order by created_at desc nulls last
    limit 2
) b on true order by author, last_author_book;
```

Важное замечание.
Подзапрос, расположенный в блоках 
- *SELECT*
- *WHERE* 
всегда имеет доступ к столбцам внешнего запроса. Подзапрос, расположенный в блоке *FROM* — только при использовании ключевого слова *LATERAL*.
