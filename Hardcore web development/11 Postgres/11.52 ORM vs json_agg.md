При работе с базой через слой *ORM*, нужно помнить о том, что *ORM* может отправлять много лишних запросов. Каждый отправленный запрос — это дополнительное время, которое будет потрачено на:
- доставку запроса по сети до сервера *Postgres*
- валидацию 
- подготовку плана исполнения запроса
- непосредственное исполнение запроса и формирование результатов
- доставку результатов запроса по сети от сервера *Postgres*


Один из подходов, позволяющих уменьшить количество отправляемых запросов:
- отправлять запросы явно, минуя слой *ORM*
- использовать функцию *json_agg* для агрегации результатов в *json*-строки
- затем, опционально, использовать полученные *json*-строки для явного создания *ORM*-объектов


```sql
SELECT
b.name AS book_name,
json_agg(distinct a.*) AS author,
json_agg(distinct c.*) AS book_category
FROM book b
LEFT JOIN book_to_author USING(book_id)
LEFT JOIN author a USING(author_id)
LEFT JOIN book_to_category USING(book_id)
LEFT JOIN book_category c USING(category_id)
GROUP by b.name
ORDER BY b.name;
```


*N.B.*
Явное создание объектов в обход ORM-слоя имеет ряд существенных недостатков:
- может сломаться *lazy loading* и другие *ORM*-фичи
- появляется необходимость вручную контролировать отношения между создаваемыми объектами
- теряется типобезопасность, валидация, миграции
- код становится менее переносимым между БД


*N.B.*
Упакованные данные занимают меньше места при передаче по сети по сравнению с неупакованными данными, передающимися в виде таблиц.

Больше места:

```
user_id | name | email        | post_id | title     | content
1       | John | john@ex.com  | 10      | Hello     | Hello world...
1       | John | john@ex.com  | 11      | Goodbye   | Goodbye world...
1       | John | john@ex.com  | 12      | Test      | Test content...
2       | Jane | jane@ex.com  | 20      | My post   | Jane's content...
2       | Jane | jane@ex.com  | 21      | Another   | More content...
```

Меньше места — данные пользователей не дублируются при передаче:

```
user_id | name | email       | posts
1       | John | john@ex.com | [{"id":10,"title":"Hello",...},  ...]
2       | Jane | jane@ex.com | [{"id":20,"title":"My post",...}, ...]
```


*N.B.*
Данные могут быть упакованы не только в *json*, но и в массив. При этом, использование дополнительной инструкции *distinct*

```sql
array_agg(DISTINCT ...) ← возвращает array
json_agg(DISTINCT ...) ← возвращает json — строку
jsonb_agg(DISTINCT ...) ← возвращает jsonb — байтовую строку
```

позволяет избежать дублирования на этапе упаковки.

*N.B.*
С помощью расширения *intarray* данные при упаковке можно дополнительно
- упорядочить
- отфильтровать
- агрегировать из нескольких источников.

```sql
CREATE EXTENSION intarray;

SELECT ARRAY[1,2,3,2,1] & ARRAY[2,3,4]; -- {2,3} -- пересечение
SELECT uniq(sort(ARRAY[1,2,3,2,1]));    -- {1,2,3} -- уникальные значения
```
