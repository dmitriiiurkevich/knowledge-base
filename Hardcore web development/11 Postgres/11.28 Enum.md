
``` sql
create type day_of_week as enum (
    'Sunday',
    'Monday',
    'Tuesday',
    'Wednesday',
    'Thursday',
    'Friday',
    'Saturday'
);
```


``` sql
create table advertiser_support_work_time (
    day_of_week day_of_week primary key,
    start_time time not null,
    end_time time not null
);
```

```sql
insert into advertiser_support_work_time 
(day_of_week, start_time, end_time) values
    ('Tuesday', '09:00:00', '19:00:00'),
    ('Thursday', '09:00:00', '19:00:00'));
```

Экономия места

```sql
select pg_column_size('воскресенье'); --> 23
select pg_column_size('воскресенье'::day_of_week); --> 4, int size
```

Проблема: при добавлении в *enum* нового значения происходят кратковременные блокировки всех таблиц, имеющих атрибуты соответствующего типа.

Поэтому лучше использовать *enum* только для тех типов данных, набор значений которых не будет никогда расширяться.

Альтернативный вариант №1 — инструкция `check`.

``` sql
create table advertiser_support_work_time (
    day_of_week varchar(11) primary key check(day_of_week in (
        'Sunday',
		'Monday',
	    'Tuesday',
	    'Wednesday',
	    'Thursday',
	    'Friday',
	    'Saturday'
    )),
    start_time time not null,
    end_time time not null
);
```

Изменение `check` проводится с менее строгими блокировками, чем изменение `enum`.

Альтернативный вариант №2 — хранить дни недели в отдельной справочной таблице.

В большинстве сценариев будет самым правильным вариантом, поскольку 
- это позволит переиспользовать эту справочную таблицу в других местах проекта,
- такую справочную таблица в будущем легко можно будет модифицировать без изменения схемы и блокировок таблиц.

Недостаток такого подхода: *join* лишней таблицы в *SQL*-запросах.
