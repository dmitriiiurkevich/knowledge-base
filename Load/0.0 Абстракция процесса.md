
### Эпоха 0

##### Batch processing machines
до1960-х


Одна программа занимает всю машину
Batch processing:
- колода перфокарт загружается в ЭВМ
- через некоторое время ЭВМ
	- или аварийно завершает выполнение программы
	- или возвращает результат в виде текста, напечатанного на бумажной ленте

Ранние компьютеры не имели периферийных устройств. А операционные системы, управлявшие их работой, были однозадачными. 

Ранние компьютеры представляли из себя т.н. пакетные системы — *batch systems* — они последовательно выполняли кванты работы, называемые заданиями — *jobs*.

Компьютер 
- загружает задание из колоды перфокарт или с магнитной ленты в собственную память.
- многократно выполняет цикл фон Неймана
	- считывает данные из памяти
	- обрабатывает считанные данные
	- записывает результаты в память
- после завершения программы выгружает на магнитную ленту  состояние своей памяти или отправляет результаты вычислений на печать

Проблема: машинное время очень дорого, а CPU простаивает во время выполнения операция ввода/вывода.

Для минимизации простоя ЭВМ хочется распараллелить два процесса:
- процесс обработки текущего задания
- процесс загрузки следующего задания

Появляется понимание того, что работа перифирийных устройств должна быть максимально изолированной от работы вычислительной части ЭВМ.

В результате периферия выделилась в отдельный класс устройств. 

В ЭВМ появились контроллеры периферии и каналы ввода-вывода — I/O channels — которые работали независимо от CPU.

CPU запускает операцию ввода/вывода → контроллер выполняет её сам → об окнчании выполнения операции контроллер сигнализирует с помощью специального прерывания.

Результат 1: 
Архитектура первых ЭВМ стала более модульной.

Результат 2:
CPU освободился от необходимости ожидания ответа от медленной периферии.

Это ещё не многозадачность, но это уже параллелизм на уровне железа.


### Две модели многозадачности

#### Кооперативная многозадачность — cooperative multitasking

Программы не полагаются на ОС в вопросах выбора момента для переключения между выполняемыми заданиями. Вместо этого в коде программ предусмотрены специальные инструкции выполнение которых приводит к передаче управления.

#### Вытесняющая многозадачность — preemptive multitasking

ОС сама решает, когда и как происходит переключение между выполняемыми процессами.

### Две проблемы кооперативной многозадачности

1.
Одна зависшая программа может повесить всю операционную систему целиком.

2.
Все программы выполняются в общем пространстве оперативной памяти.

Суть проблемы:
- Несколько программ выполняются прарллельно, используя общую память.
- Одна из программ сохраняет свои данные в память после чего приостанавливает своё исполнение. 
- Другая программа во время своего исполнения случайно или намеренно может использовать те же ячейки памяти для хранения своих данных.
- Данные, сохранённые первой программой, повреждены.
- При попытке возобновить своё исполнение первая программа вылетает с ошибкой.

Ну и не стоит забывать, что общая память — это очень и очень небезопасный способ хранения конфиденциальных данных.

##### Варианты решения проблем

2.1.
Память заранее делится на диапазоны, затем каждая программа компилируется под конкретный диапазон адресов — компилятор генерировал код с абсолютными адресами. Хочешь запустить программу в другом дипазоне адресов памяти — перекомпилируй. Оператор ошибся при распределении диапазонов между программами — вся система зависла.

2.2.
Memory relocation — компилятор прикладывает к исполняемому файлу программы таблицу релокаций — смещений относительно некоторого базового адреса. На основе этой таблицы загрузчик вычисляет реальные адреса в момент запуска программы. Теперь скомпилированнцю программу можно загрузить куда угодно, но только один раз при загрузке. Переместить работающую программу по-прежнему нельзя.

2.3.
Swapping — в каждый момент времени в памяти находится только одна программа, все остальные остальные — на диске. Переключение между программами занимает чудовищное время.


### Процесс

#### Что такое процесс?

Самый распространённый ответ: процесс — это программа в момент исполнения — *program in execution*.

И этот ответ не вполне верен. Программа состоит из 
- исполняемого файла — *executable file*
- библиотек — *libraries* 

Каждая библиотека также является программой. Код библиотек связан с кодом исполняемого файла (а также с кодом других библиотек) с помощью зависимостей — *dependencies*.

Таким образом, с одной стороны, один процесс может состоять из нескольких одновременно исполняемых программ. С другой стороны, каждая программа может одновременно выполняться в нескольких параллельных процессах.

Более правильное определение: процесс — это виртуальная среда, создающаяся операционной системой для выполнения 
- программы
- части программы
- или нескольких совместно выполняющихся программ.


#### Процесс изнутри

Фактически процесс состоит из двух участков памяти:
- user space
- участок памяти kernel space




### Kernel mode / user mode + Kernel space / user space + Kernel code / user code


Существуют два режима работы CPU:
- kernel mode
- user mode

Текущий режим работы процессора определяется битом, который хранится в поле CPL (Current Privilege Level) процессорного регистра CS (Code Segment).

И существуют области памяти:
- kernel space — один общий раздел
- user space — по одному отдельному разделу на каждый процесс











##### Что меняется в зависимости от режима?

1\.
Доступ к памяти




- kernel mode: процессор может читать/писать любую память (включая kernel space)
- user mode: процессор может читать/писать только user space.

2\.
Инструкции процессора
- kernel mode: процессор может выполнять привилегированные инструкции (например, изменение CR3, работа с I/O портами)
- user mode: попытка выполнения процессором привилегированных инструкций вызовет исключение.


##### Как происходит переключение режимов?

С помощью системных вызовов:
- `syscall`  → kernel mode
- `sysret` → user mode

Процесс не знает и не контролирует, в каком режиме он находится в тот или иной момент. Процесс просто выполняет инструкции.

Когда процесс делает системный вызов, CPU автоматически переключается в kernel mode и начинает выполнять код из kernel space.

Когда выполнение системного вызова завершается, CPU автоматически переключается обратно в user mode.

Процесс не может сам переключить процессор в kernel mode, только через системный вызов или обработку прерывания.

Процесс не может узнать, в каком режиме сейчас находится процессор, поскольку режим выполнения — это внутреннее состояние процессора.




### Kernel code

Код ядра — это просто набор функций в памяти. Нет никакого отдельного ядра, которое работает параллельно пользовательским процессам.

Как CPU обрабатывает системный вызов?

1\.
CPU переключается в kernel mode и переходит на конкретный адрес в памяти, по которому лежит функция-обработчик сооветствующего системного вызова

2\.
CPU выполняет эту функцию — код ядра.

3\.
После заверщения выполнения функции CPU переключается в user mode и продолжает выполнение программы, совершившей системный вызов.

##### Может ли произойти переключение процессов, если текущий процесс выполняет код ядра?

Да, может.

1\.
Процесс A делает системный вызов `read()` для того, чтобы прочитать данные из файла.

2\.
CPU переходит в kernel mode, начинает выполнять функцию ядра `sys_read()`.

3\.
Функция `sys_read()` понимает, что данных нет в кеше, а следовательно, они будут готовы ещё не скоро.

4\.
Функция вызывает `schedule()` — планировщик ОС.

5\.
Планировщик помечает процесс A как WAITING — ожидающий I/O — и выбирает для выполнения другой процесс B.

6\.
Происходит context switch и начинает выполняться процесс B.

7\.
Когда данные готовы, приходит прерывание от контроллера диска. Обработчик прерывания помечает процесс A как READY.

9\.
В какой-то момент выберет для исполнения процесс A.

10\.
Процесс A продолжит выполнение внутри функции `sys_read()` — дочитает данные, и только после этого CPU переключится обратно в user mode.

Немного истории

Ранние компьютеры не имели периферийных устройств. А прото-операционные системы, управлявшие работой первых компьютеров, были однозадачными. 

Ранние компьютеры были пакетными системами — *batch systems* — которые последовательно выполняли кванты работы, называемые заданиями — *jobs*.

Компьютер загружал задание с магнитной ленты в собственную память.

Затем компьютер многократно выполнял цикл фон Неймана:
- считать данные из памяти
- обработать данные
- записать результаты в память.

После завершения выполнения программы компьютер выгружал на магнитную ленту состояние своей памяти, содержащее результаты выполнения программы.

70-е годы

Уже тогда — во времена магнитной ленты и перфокарт — проявилась первая проблема, для решение которой потребовалась бы та или иная реализации многозадачности: компьютеров было мало, процессорное время стоило дорого, загрузка программ и выгрузка результатов их выполнения занимала существенное время, в течение которого дорогостоящий процессор простаивал. 

80-е годы

Шло время, компьютеры постепенно становились всё более *персональными*. 

Тренд №1
С одной стороны, стоимость процессорного времени всё время снижалась, с другой стороны, у компьютеров стала появлялась всё более развитая периферия — внешние накопители, сетевые карты, и т. п. 

Реализованная традиционным образом программа отправляет пакет через сетевую карту, или запрос на считывание блока данных с внешнего накопителя, а затем приостанавливает своё выполнение вплоть до получения ответа от сетевой карты или внешнего накопителя. Это вновь приводит к неэффективному использованию ресурсов процессора, хотя теперь быстродействие выполняемых пользователем программ становится важнее стоимости процессорного времени. 

Тренд №2
Пользователи персональных компьютеров — уже не инженеры, а самые обычные люди — внезапно захотели слушать музыку одновременно с работой в текстовом редакторе.

Начало 90-х.

*MS-DOS* — самая популярная, но всё ещё однозадачная операционная система. Для того, чтобы запустить следующую программу, пользователям *MS-Dos* сначала было необходимо завершить выполнение предыдущей программы.

Windows 3.1 — операционная система, в которой на основе новой базовой абстракции — *процесса* — реализована так называемая *кооперативная многозадачность*:
- все программы загружаются в общую память
- программа приостанавливает своё выполнение и передаёт управление следующей программе в тот момент, когда сама посчитает нужным.

Проблема №1
Все программы используют общую память, а потому одновременно выполняющиеся имеют возможность навредить друг другу, что небезопасно и может приводить к зависаниям.

Проблема №2 
Если зависнет хотя бы одна из выполняющихся программ, вместе с ней сразу зависнет и вся операционная система.

Середина 90-х.

Появляются новые операционные системы (*Windows 9x*, *NT*, *Linux*) с программной поддержкой технологии *виртуализации памяти*, позволяющей реализовать новый тип многозадачности – _вытесняющую многозадачность_.

С появлением процессоров *Pentium* персональные компьютеры получают аппаратную поддержку технологии *виртуализации памяти* — *MMU* — *Memory Management Unit*.

Теперь каждой запущенной программе операционная система выделяет *виртуальную среду выполнения* — своего рода контейнер, снабжённый виртуальным адресным пространством — *virtual memory space* — виртуальным пространством адресов памяти, доступных для использования этой программой.

В то же время в самой операционной системе реализованы следующие механизмы:
- механизм трансляции виртуальных адресов памяти из *virtual memory space* в реальные адреса физической памяти
- механизм аллокации и деаллокации физической памяти для нужд программ, выполняющихся в виртуальных средах
- механизм распределения ресурсов процессора между программами, выполняющимися в виртуальных средах
- 
Эти виртуальные среды выполнения и называются *процессами* операционной системы.

При этом,
- у каждого процесса создаётся иллюзия того, что он является единственным процессом, исполняемым в операционной системе, и что ему доступно всё адресное пространство физической памяти целиком
- ни один процесс почти никогда не может обращаться к какой-либо ячейке памяти напрямую — по её физическому адресу.
- ни один процесс не может напрямую обращаться ни к какому другому процессу, посредником в общении между *процессами* выступает операционная система
- ни один процесс не может напрямую обращаться ни к одному физическому устройству, посредником в общении между процессами и физическими устройствами также выступает операционная система.

В каждый момент времени 
- один процесс выполняется
- все остальные процессы — спят. 

В операционных системах появляется специальный процесс — планировщик —который раз в 20 миллисекунд
- просыпается раз в 20 миллисекунд
- выбирает один из спящих процессов
- будит его на следующие 20 миллисекунд
- засыпает сам.

Такой вид многозадачности называется _вытесняющей многозадачностью_.
