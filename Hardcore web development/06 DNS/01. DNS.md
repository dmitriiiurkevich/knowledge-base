
## DNS as a distributed server system
*Domain Name System* — система доменных имён — чаще всего используется для разрешения доменных имён, то есть, для преобразования: имя доменной зоны + имя хоста в этой доменной зоне → *IP*-адрес этого хоста. Хотя, такое преобразование — это далеко не единственная функция *DNS*.

www.google.com — это *host name* → www + google.com. — где
- google.com. — это *domain name* — полное доменное имя
- www — это имя сервера, на котором размещён сайт, в этом домене. 

В одном и том же домене помимо сервера с именем *www* может быть множество разных других серверов с разными именами и разными IP-адресами: 
- mail.google.com
- maps.ggogle.com
- drive.google.com
- ...

Максимальный размер *fully qualified domain name* (*FQDN*) = 255 байт, с ограничением в 63 байта на каждое имя домена.

### Delegation authority
Всё пространство имён разделено на отдельные части — *домены*. Ответственность за каждый домен делегирована отдельной организации. 

Для повышения надёжности все *DNS*-сервера распределены и многократно дублируются.

### Зачем это нужно?
Доменные имена более привычны людям.

*DNS* позволяет владельцу доменного имени изменять свою сетевую инфраструктуру. Если со временем *IP*-адреса серверов того или иного домена изменятся, то владельцу соответствующего домена, достаточно будет просто изменить *DNS*-запись.

*DNS* позволяет владельцу доменного имени расширять свою сетевую инфраструктуру при увеличении нагрузки. Одно и то же доменное имя могут обслуживать несколько серверов с разными *IP*-адресами.

### Как это работало раньше?
Изначально все ресурсы сети ARPAnet имели только числовые идентификаторы. И это  вызывало неудобства уже тогда.

Потом появился файл hosts, который 
- хранил соответствия: имя ресурса → *IP*-адрес ресурса
- лежал на *FTP Network information center of Stanford university*
- весил несколько килобайт, но постепенно разросся до нескольких мегабайт
- постоянно обновлялся
- выкачивался по каналу, скорость которого не превышала нескольких тысяч бод (бит/сек). 

И в некотором рудиментарном виде файл hosts до сих пор остаётся с нами
- `/etc/hosts`
- `C:\Windows\System32\drivers\etc\hosts`

Решение придумал Dr. John Postel (*RFC 1034*, *RFC 1035*), и со временем это решение развилось в *DNS*, которую мы используем до сих пор.
### Как это работает сейчас?
*DNS* — это распределённая иерархическая система серверов.

## DNS as a network protocol
*DNS* — это протокол прикладного уровня. Работает как поверх *UDP*, так и поверх *TCP*. 

Default port = 53

*UDP* используется для простых *DNS*-запросов.

*TCP* используется
- в случае, если ответ на *DNS*-запрос слишком велик для передачи респонса в одном *UDP*-сообщении (ограничение составляет 512 байт для стандартного *DNS* и до 4096 байт для *DNS* через *EDNS* — механизмы расширения для *DNS*), *DNS*-сервер установит флаг *Truncation* в респонсе, возвращаемом клиенту, после чего клиент повторит свой запрос по протоколу *TCP*.
- для репликации *DNS*-серверов — поскольку в процессе репликации *DNS*-сервера обмениваются данным о целых доменных зонах, this can involve large amounts of data and requires the reliability of the *TCP*. 
- некоторые клиенты, постоянно отправляющие *DNS*-запросы, могут использовать для этого протокол *TCP*, поддерживая постоянное *TCP*-соединение (*keep-alive connection*).
- в протоколе *DNSSEC* — *DNS Security Extensions* — который использует цифровые подписи для защиты от *DNS spoofing* атак.

## DNS-записи
На *DNS*-серверах хранятся так называемые *DNS*-записи (или *Resource Records* (*RR*). Каждая *DNS*-запись имеет:
- тип записи — для чего предназначена запись
- класс записи — в каких сетях используется запись (почти всегда это *IN* — *Internet*)



![[All_active_dns_record_types.png]]

Типов *DNS*-записей очень много, рассмотрим самые основные.

### A, AAAA — Address record
Записи, которые используются для определения *IP*-адреса хоста, имеют тип
- *A* — для *IPv4*-адресов
- *AAAA* — для *IPv6*-адресов

У домена может не быть ни *A*, ни *AAAA* записи, если, например, домен создавался только для того, чтобы поднять в нём почтовый или какой-нибудь иной сервер.

``` 
$ host -t A google.com
google.com has address 172.217.17.142
``` 

``` 
$ host -t A yandex.ru
yandex.ru has address 77.88.44.55
yandex.ru has address 77.88.55.88
yandex.ru has address 5.255.255.77
``` 

Если у одного домена окажется несколько *A* записей, то, скорее всего
- это следствие того, что между указанными серверами происходит ротация запросов с использованием *round-robin*.
- список серверов будет возвращаться в случайном порядке 


`$ host -t AAAA google.com`
`google.com has IPv6 address 2a00:1450:4017:811::200e`

`$ host -t AAAA yandex.ru`
`yandex.ru has IPv6 address 2a02:6b8:a::a`

 *Round-robin* плохо работает с *IPv6* адресами.
 
### MX — Mail eXchange record
Записи, которые используются для определения *IP*-адреса почтового сервера. Например, для домена *gmail.com* существует несколько *MX* записей, со следующими значениями: 

| приоритет 🠙 | доменное имя сервера    |
| ------------ | ----------------------- |
| 1            | aspmx.l.google.com      |
| 5            | alt1.aspmx.l.google.com |
| 5            | alt2.aspmx.l.google.com |
| 10           | alt3.aspmx.l.google.com |
| 10           | alt4.aspmx.l.google.com |

```
$ host -t MX google.com
google.com mail is handled by 10 smtp.google.com.
```

### CNAME — Canonical Name record
Определяет псевдоним для какого-то другого доменного имени.

Для одного и того же *IP*-адреса можно задать несколько доменных имён. Есть два варианта того, как это можно сделать:
- задать несколько A-записей, которые указывают на один *IP*-адрес
- задать одну A-запись и несколько *CNAME*-записей, ссылающихся на неё

Если у домена есть *CNAME*-запись, то у этого домена не может быть ни *A*-записи, ни *MX*-записи, ни каких-либо других *DNS*-записей.

Не рекомендуется делать *CNAME*-записи, которые ссылались бы на другие *CNAME*-записи.

```
$ host -t CNAME search.google.com
search.google.com is an alias for www3.l.google.com.
``` 
 
### SRV — Service record
Записи, которые используются для определения *IP*-адреса хоста и номера порта, на котором развёрнут тот или иной сервис.

Например, пусть где-то в домене *mydomain.com* работают два *Jabber*-сервера: основной и резервный. Тогда команда

```
$ host host -t SRV _xmpp-server._tcp.example.com
```

вернёт две *DNS* записи типа *SRV*, например, такие:

| приоритет 🠙 | вес | порт | доменное имя сервера     |
| ------------ | --- | ---- | ------------------------ |
| 0            | 5   | 5269 | xmpp.mydomain.com        |
| 20           | 0   | 5269 | backup_xmpp.mydomain.com |
 
 Приоритет в обратном порядке. Вес используется для распределения нагрузки между серверами, которые имеют один и тот же приоритет.

### NS — Name Server record
Единственный тип *DNS*-записи, без которой не может обойтись ни один домен, поскольку она содержит имена *DNS*-серверов, обслуживающих этот домен. 

Предположим, вы являетесь владельцем домена *example.com*. И предположим, вы хотите 
- или поднять в вашем домене ваш собственный *DNS*-сервер, и использовать его в качестве *Authoritative nameserver* для вашего домена
- или делегировать обслуживание *DNS*-запросов, связанных с вашим доменом, сторонней организации, чтобы *DNS*-запросы, связанные с вашим доменом, выполнялись бы на её серверах.  

Вам необходимо на вышестоящий *DNS*-сервер — в данном случае, на *TLD-nameserver*, отвечающий за доменную зону *com* — поместить специальную *NS*-запись, которая будет содержать имена ваших *DNS*-серверов, например: 
- *ns1.mydomain.com* 
- *ns2.mydomain.com*.

После чего именно эти сервера становятся первоисточником информации о вашем домене,  из которого эта информация начнёт распространяться и кэшироваться по всем остальным *DNS*-серверам.

К *NS*-записям могут прилагаться так называемые "приклеенные" *A*-записи — *glue records* — позволяющие определить не только доменные имена, но и *IP*-адреса ваших *DNS*-серверов.

[[#Delegation authority]] — это процесс, результатом которого в любом из двух вышеописанных случаев является появление *NS*-записей вашего домена на вышестоящих (в иерархическом смысле) *DNS*-серверах.

### PTR — Pointer record
Все  *DNS*-зоны кроме одной — это т. н. прямые (*direct*) зоны, в них осуществляется прямое преобразование доменного имени в *IP*-адреса хоста. 

Кроме того существует одна специальная реверсивная (*reverse*)  *DNS*-зона — in-addr.arpa — в которой 
- содержатся *DNS*-записи записи типа PTR
- осуществляется обратное преобразование — *IP*-адрес хоста преобразуется в доменное имя этого хоста. 

IP-адреса, записанные в обратном порядке, представляются в виде доменных имён в этой зоне:

77.88.55.66 → 66.55.88.77.in-addr.arpa. → *DNS* → *PTR*-record → domain name

### TXT — Pointer record
С помощью таких *DNS*-записей владелец домена может подтвердить право обладания своим доменом. 

Например, владелец домена хочет поднять в своём домене почтовый сервер *Gmail* для обработки своей корпоративной почты. Или какой-нибудь другой сервис *Google* для каких-нибудь других целей.

Владелец домена через специальный интерфейс отправит в *Google* соответствующую заявку. Прежде чем выполнить полученную заявку, *Google* проверит, что отправитель заявки действительно является владельцем домена. Для этого *Google*-сервис сгенерирует уникальную строку и попросит автора заявки разместить её в новой *TXT*-записи на соответствующем *DNS*-сервере.  

...

There are 4 types of *DNS*-servers involved in loading a webpage:
1. *DNS-recursor*
2. *root nameserver*
3. *TLD-nameserver*
4. *authoritative nameserver*

### DNS-recursor
*DNS-recursor* — *DNS-recursive resolver* — a server designed to receive *DNS* queries from client machines. По сути своей является кеширующим прокси-сервером для *DNS*-запросов.

After receiving a *DNS* query from a web client, a recursive resolver 
- will either respond with cached data, 
- or will send a requests 
	- to a *root nameserver*, 
	- then to a *TLD-nameserver*, 
	- finally to an *authoritative nameserver*
- and finally will return an *IP* address (or several ones), or probably even a pair: *IP* address + port. 

After receiving a response from the *authoritative nameserver* containing the requested *IP*-address, the *recursive resolver* then sends a response to the client.

*DNS*-резолверы обычно являются частью сетевой инфраструктуры, предоставляемой *Internet Service Provider* (*ISP*) или *Cloud Service Provider* (*CSP*).

Роутер получает адрес *DNS*-резолвера, предоставляемого провайдером, одновременно с получением собственного *IP*-адреса по протоколу *DHCP*. В дальнейшем этот адрес можно всегда изменить.

Также существуют публичные *DNS*-сервисы, такие как 
- *Google DNS* — самый популярный в мире, 
- *Cloudflare DNS* 
- *OpenDNS*. 

Свой собственный *DNS-resolver* можно развернуть локально, например — *dnsmasq.*

В файле `etc/resolv.conf` хранятся настройки, определяющие то, как ваша операционная система взаимодействует с *DNS*.

### Root nameserver
Широко распространённый миф: в мире существует ровно 13 корневых *DNS*-серверов. В действительности существует не 13 серверов, а 13 систем корневых *DNS*-серверов (*root DNS server systems*), которые называются 
- a.root-servers.net
- b.root-servers.net
- c.root-servers.net
	⁝
- m.root-servers.net

Каждая система корневых *DNS*-серверов представляет из себя множество физических хостов-реплик, находящихся в различных локациях сети *Internet* и имеющих — благодаря применению технологии *anycast addressing* — один общий *IP*-адрес. Таким образом достигается малое время отклика и стабильная работа всей системы.

Корневые *DNS*-серверов управляются двенадцатью различными организациями, действующими на основании соглашений с корпорацией *ICANN*.

Любой *DNS-recursor* знает *IP*-адреса всех 13-ти корневых *DNS*-серверов.

*DNS-recursor* отправляет корневому *DNS*-серверу запрос, содержащий доменное имя.

Корневой *DNS*-сервер проверяет, содержится ли это доменное имя в его кэше. И если нет, то извлекает домен верхнего уровня (*.com*, *.net*, *.org*, ...) из полученного доменного имени и возвращает *DNS-recursor*'у адрес соответствующего *TLD-nameserver*'а.

Аналогичным образом, если вы пытаетесь дать себе ответ на вопрос: почему не работает ваш домен, то одним из первых шагов на пути диагностики должен стать опрос каждого из этих *DNS*-серверов, чтобы установить, содержит ли он хоть какие-нибудь *NS*-записи вашего домена.

Не каждый *DNS*-запрос обрабатывается корневым сервером. По разным оценкам, только от 18 до 32 % разрешений доменных имён приводит к обращению непосредственно к одному из корневых серверов, остальные запросы используют кэшированные *DNS*-записи.

### TLD-nameserver
*TLD-nameserver* — *Top Level Domain nameserver* — хранит информацию обо всех доменах второго уровня, существующих в соответствующей доменной зоне первого уровня. По запросу возвращает адрес *Authoritative nameserver* для соответствующего домена второго уровня.

### Authoritative nameserver
Непосредственно на этих серверах обычно хранятся *DNS*-записи типов *A (AAAA)*, *MX*, *SRV*,... , относящиеся к конкретному домену второго уровня.

А также записи типа *CNAME*, в которых хранятся псевдонимы, содержащие ссылки на другие имена того же домена второго уровня. 

Если вместо *IP*-адреса, содержащегося в записи типа *A*, *DNS*-резолверу вернётся псевдоним, содержащийся в записи типа *CNAME*, то *DNS*-резолвер продолжит процедуру поиска. Как правило, для продолжения поиска *DNS*-резолверу не нужно повторно обращаться к *root nameserver*, следующий запрос после получения *CNAME*-записи *DNS*-резолвер отправляет на тот же *authoritative nameserver*.

## What are the steps in a DNS lookup?
Often, DNS lookup information is cached either locally inside the querying computer or remotely in the DNS infrastructure. There are typically 8 steps in a DNS lookup. When DNS information is cached, steps are skipped from the lookup process, which makes it quicker. The example below outlines all 8 steps when nothing is cached.

### The 2 steps prior to start a DNS lookup:
1. The browser checks whether a value of the *DNS* record that being requested for is already stored in its cache. And whether the value is out of date.

2. The browser checks whether a value of the *DNS* record that being requested for is already stored in *OS* cache. And whether the value is out of date. This cache is managed by the operating system, separate from any network or local DNS servers or resolvers. Whenever a *DNS* request is made, the operating system checks this cache first before contacting a *DNS* server. This is to save time and resources by using previously obtained *DNS* resolution results.

It's also possible to disable *DNS* caching at the browser or the *OS* level.

### The 8 steps in a DNS lookup:
1. A user types *example.com* into a web browser and the query travels into the *Internet* and is received by a *DNS recursive resolver*.
2. The resolver then queries a *DNS root nameserver*.

	``` 
	$ host -v -t NS com. i.root-servers.net
	Trying "com"
	Using domain server:
	Name: i.root-servers.net
	Address: 192.36.148.17#53
	Aliases:
	
	;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 35719
	;; flags: qr rd; QUERY: 1, ANSWER: 0, AUTHORITY: 13, ADDITIONAL: 15
	
	;; QUESTION SECTION:
	;com.                           IN      NS
	
	;; AUTHORITY SECTION:
	com.                    172800  IN      NS      k.gtld-servers.net.
	com.                    172800  IN      NS      m.gtld-servers.net.
	com.                    172800  IN      NS      b.gtld-servers.net.
	com.                    172800  IN      NS      c.gtld-servers.net.
	com.                    172800  IN      NS      j.gtld-servers.net.
	com.                    172800  IN      NS      i.gtld-servers.net.
	com.                    172800  IN      NS      d.gtld-servers.net.
	com.                    172800  IN      NS      g.gtld-servers.net.
	com.                    172800  IN      NS      h.gtld-servers.net.
	com.                    172800  IN      NS      f.gtld-servers.net.
	com.                    172800  IN      NS      e.gtld-servers.net.
	com.                    172800  IN      NS      a.gtld-servers.net.
	com.                    172800  IN      NS      l.gtld-servers.net.
	```

3. The *root server* then responds to the *resolver* with the address of a *Top Level Domain DNS server* (such as *.com* or *.net*) — or *TLD* — which stores the information for its domains. When searching for *example.com*, our request is pointed toward the *.com* *TLD*.
4. The *resolver* then makes a request to the *.com* *TLD*.

	```
	$ host -v -t NS google.com f.gtld-servers.net.
	Trying "google.com"
	Using domain server:
	Name: f.gtld-servers.net.
	Address: 192.35.51.30#53
	Aliases:
	
	;; ->>HEADER<<- opcode: QUERY, status: NOERROR, id: 63497
	;; flags: qr rd; QUERY: 1, ANSWER: 0, AUTHORITY: 4, ADDITIONAL: 8
	
	;; QUESTION SECTION:
	;google.com.                    IN      NS
	
	;; AUTHORITY SECTION:
	google.com.             172800  IN      NS      ns2.google.com.
	google.com.             172800  IN      NS      ns1.google.com.
	google.com.             172800  IN      NS      ns3.google.com.
	google.com.             172800  IN      NS      ns4.google.com.
	
	;; ADDITIONAL SECTION:
	ns2.google.com.         172800  IN      AAAA    2001:4860:4802:34::a
	ns2.google.com.         172800  IN      A       216.239.34.10
	ns1.google.com.         172800  IN      AAAA    2001:4860:4802:32::a
	ns1.google.com.         172800  IN      A       216.239.32.10
	ns3.google.com.         172800  IN      AAAA    2001:4860:4802:36::a
	ns3.google.com.         172800  IN      A       216.239.36.10
	ns4.google.com.         172800  IN      AAAA    2001:4860:4802:38::a
	ns4.google.com.         172800  IN      A       216.239.38.10
	
	Received 276 bytes from 192.35.51.30#53 in 49 ms
	```

5. The *TLD* server then responds with the *IP* address of the domain’s *nameserver*, *example.com*.
6. Lastly, the *recursive resolver* sends a query to the domain’s *nameserver*.
7. The *IP* address for *example.com* is then returned to the *resolver* from the *nameserver*.
8. The *DNS* resolver then responds to the web browser with the *IP* address of the domain requested initially.

N.B.: If the cache of the *DNS* resolvers or cache of a *TLD* server stores a value of the requested record, and that value has not become outdated, the *DNS* lookup stops.

Once the DNS lookup have returned the *IP* address for *example.com*, the browser is able to make the request for the web page:
9. The browser makes a *HTTP* request to the *IP* address.
10. The server at that *IP* returns the webpage to be rendered in the browser.

## What are the types of DNS queries?
The *resolver* starts the sequence of *DNS* queries. In a typical *DNS* lookup three types of queries occur:
1. *recursive query*
2. *iterative query*
3. *non-recursive query*

### Recursive DNS query
In a recursive query, a *DNS* client requires that a *DNS* server (typically a *DNS recursive resolver*) will respond to the client with 
- either the requested resource *DNS record* 
- or an error message if the resolver can't find the *DNS record*.

### Iterative DNS query
In this situation the *DNS* client will allow a *DNS* server to return the best answer it can.

If the queried *DNS* server does not have a match for the query name, it will return a referral to a *DNS* server authoritative for a lower level of the domain namespace. The *DNS* client will then make a query to the referral address. This process continues with additional *DNS* servers down the query chain until either an error or timeout occurs.

### Non-recursive DNS query
This typically occurs when a *DNS* resolver client queries a *DNS* server for a record that it has access to 
- either because it's authoritative for the record 
- or because the record exists inside of its cache.

Typically, a *DNS* server caches *DNS* records to prevent additional load on upstream *DNS* servers.

## DNS caching

### Browser DNS caching
When a request is made for a *DNS* record, the browser cache is the first location checked for the requested record.

In Chrome, you can see the status of your DNS cache by going to chrome://net-internals/#dns.

### OS level DNS caching
The operating system level *DNS* resolver is the second and last local stop before a *DNS* query leaves your machine.

The process inside your operating system that is designed to handle this query is commonly called a *DNS stub resolver* or *DNS client*.

When a *stub resolver* gets a *DNS* request from an application, it first checks its own cache to see if it has the record. If it does not, it then sends this *DNS* query (with a *recursive* flag set), outside the local network to a *DNS recursive resolver* inside the *Internet* service provider *(ISP)*.

When the *recursive resolver* inside the *ISP* receives a *DNS* query, like all previous steps, it will also check to see if the requested host → *IP*-address translation is already stored inside its local persistence layer.

The *recursive resolver* also has additional functionality depending on the types of records it has in its cache:
#### 1.
If the *resolver* does not have the *A* record, but does have the *NS* record for the appropriate *authoritative nameserver*.

Then the *resolver* will query those *authoritative nameserver* directly, bypassing several steps in the *DNS* query. 

#### 2.
If the *resolver* does not have the *NS* record. 

Then the *resolver* will send a query to the appropriate *TLD name servers*, skipping the *root name server*.

#### 3.
If the *resolver* does not have records pointing to the *TLD name servers*.

Then the *resolver* query one of the *the root name servers*. This event typically occurs after a *DNS* cache has been purged.

## TTL
*TTL* — *time to live* — это теоретическое время кэширования ваших *DNS*-записей на, вообще говоря, сторонних *DNS*-резолверах.

```
$ host -v -t NS google.com f.gtld-servers.net.
...
google.com.             172800  IN      NS      ns2.google.com.
...
```

В данном случае, *TTL* = 172800 секунд ⬏

Маленькие значения *TTL* нехороши тем, что они увеличивают 
- нагрузку на *DNS*-сервера и резолверы
- время загрузки страницы в случае, если ваши *DNS*-сервера находятся на одном континенте, а *DNS*-резолверы ваших пользователей — на другом.

Большие значения *TTL* нехороши тем, что они увеличивают время существования outdated кэшей. Впрочем, где-то в далёкой глубинке администраторы *DNS*-резолверов могут завышать на своих серверах значение параметра *minimum TTL* для того, чтобы их не слишком мощные *DNS*-резолверы не совершали бы лишней работы. 

Завышать могут вплоть до месяца. Хотя, обычное значение *TTL* ≈ 1 час.

## Localhost
В *A*-записи может быть указан в том числе и локальный *IP*-адрес. В этом случае пользователь, разрезолвивший доменное имя, действительно зайдёт на свой собственный хост по *http*.

Заменить *IP*-адрес в *A*-записи вашего домена на 127.0.0.1 или 192.168.02 может быть хорошей идеей при отражении *DDoS*-атаки. В результате такой замены *botnet* начинает атаковать сам себя, и по крайней мере часть системных администраторов заражённых хостов может быстро обнаружить подозрительный трафик.

Заменить *IP*-адрес в *A*-записи вашего домена на *IP*-адрес *Google* или *NASA* не может быть хорошей идеей при отражении *DDoS*-атаки.

## Безопасность DNS 
Протокол *DNS* разрабатывался очень давно, поэтому с безопасностью там всё очень плохо.

### Чужой DNS-резолвер с поддельными записями  
Всё ещё актуальная проблема. Инвалидация *DNS*-записей может использоваться локальными провайдерами для блокировки доступа к сайтам.

Также возможно перенаправление запросов к 8.8.8.8 — *Google DNS* — во внутреннюю сеть провайдера. Это скорее всего нарушает договор о предоставлении доступа в *Internet*, заключённый между клиентом и его провайдером, но кто же это заметит?

### UDP response forgery
Атака на *DNS*-клиента
- клиент отправляет DNS-запрос куда-то в сеть
- в ответ откуда-то из сети приходит *UDP*-пакет с невалидным *IP*-адресом (а протокол *UDP* не позволяет достоверно установить, откуда именно тот или иной пакет был отправлен)
- клиент использует невалидный *IP*-адрес и оказывается на сайте злоумышленников

### Cache poisoning
Атака на *DNS*-resolver, аналогичная *UDP response forgery*
- резолвер отправляет запрос на сервер, на котором, как он ожидает, хранится *DNS*-кэш
- в ответ приходит *UDP*-пакет с невалидным *IP*-адресом и огромным *TTL* 
- резолвер пересылает невалидный  *IP*-адрес клиенту
- клиент использует невалидный *IP*-адрес и оказывается на сайте злоумышленников

### DNS Amplification
Уже не самый актуальный способ проведения *DDoS*-атак.

На плохо настроенный *DNS*-сервер приходит *UDP*-пакет, якобы отправленный с *IP*-адреса одного из серверов жертвы, с просьбой разрезолвить доменное имя какого-нибудь сервера, у которого есть сотня *A*-записей. В результате чего в жертву летит несколько килобайт текста.

Именно с использованием этой уязвимости происходили самые крупные *DDoS*-атаки в районе 2012—2013 годов. Помимо отказа в обслуживании такая атака может существенно увеличить стоимость использования арендованного в облаке сервера.

N.B.: *DNS*-трафик не шифруется, что позволяет провайдерам анализировать *DNS*-запросы клиентов. Поэтому, если вы поднимаете *VPN*, чтобы обходить блокировки, то не забывайте шифровать трафик в том числе и до *DNS*-сервера. Иначе можно спалиться.
